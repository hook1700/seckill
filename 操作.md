## 1. 架构与数据流

```
┌──────────────┐
│   Client     │
└──────┬───────┘
       │ HTTP
┌──────▼───────┐
│  Gin (Go)    │
│  API Server  │
└──┬────────┬──┘
   │        │ LPUSH         Producer
   │        ▼
┌───┴───┐  ┌──────────────┐
│ Redis │  │   Kafka      │
│ Stock │  │ seckill_order│
└───────┘  └──────┬───────┘
                  │ Consumer Group
                  │ (seckill-group)
                  ▼
           ┌──────────────┐
           │    MySQL     │
           │  orders 表   │
           └──────────────┘
```

**数据流说明：**

| 步骤 | 组件             | 说明                             |
|----|----------------|--------------------------------|
| 1  | Client         | 用户发起秒杀请求                       |
| 2  | Gin API Server | Go 后端接收 HTTP 请求                |
| 3  | Redis          | 预减库存（LPUSH），快速响应               |
| 4  | Kafka          | 异步消息队列，削峰填谷                    |
| 5  | Consumer Group | Go 消费者组 `seckill-group` 消费订单消息 |
| 6  | MySQL          | 持久化订单到 `orders` 表              |

## 3. Kafka Topic 管理

### 3.1 创建 Topic

```bash

  --bootstrap-server kafka:9092 \
  --create --topic seckill_order \
```

## 5. 常见问题排查

### 问题 1：Kafka 容器反复退出（Exit Code 137）

**现象**

```bash
docker ps -a | grep kafka
# Exited (137) 5 minutes ago
```

**排查过程**

```bash
# 1. 检查是否 OOM
docker inspect seckill_kafka --format='{{.State.OOMKilled}}'
# 输出 true

# 2. 查看系统内存
free -m

# 3. 查看内核日志
dmesg -T | grep -i 'oom\|killed process'
```

**根因**

- Kafka（Confluent Platform）默认 JVM 堆内存需求大
- 云服务器内存不足（1~2GB）
- 同时存在 MySQL + Redis + Kafka + ZooKeeper，内存超限被 Linux OOM Killer 强制终止

**解决方案**

在 `docker-compose.yaml` 中限制各服务资源：

```yaml
# Kafka JVM 内存限制
environment:
  KAFKA_HEAP_OPTS: "-Xms128M -Xmx384M"
  KAFKA_JVM_PERFORMANCE_OPTS: "-XX:MetaspaceSize=48m -XX:+UseG1GC"

# ZooKeeper 内存限制
environment:
  JVMFLAGS: "-Xms64M -Xmx128M"

  # 减少 Kafka 网络/IO 线程数
  KAFKA_NUM_NETWORK_THREADS: 2
  KAFKA_NUM_IO_THREADS: 2
```

### 问题 2：kafka-console-consumer 没有输出

**现象**

```bash
docker exec -it seckill_kafka kafka-console-consumer \
  --bootstrap-server kafka:9092 \
  --topic seckill_order \
  --from-beginning
# 无任何输出
```

**排查过程**

```bash
# 1. 确认 Topic 是否存在
kafka-topics --list
# 输出包含 seckill_order ✅

# 2. 查看 Consumer Group
kafka-consumer-groups --list
# 发现 seckill-group 已存在

# 3. 查看 Kafka 日志
docker logs seckill_kafka --tail 50
# 发现 seckill-group 已稳定消费
```

**根因**

- Go 服务（`seckill-group`）已经消费完所有消息
- `console-consumer` 使用不同的 `group.id`，但消息已被消费且未保留
- Kafka 日志保留策略可能已清理旧消息

**解决方案**

```bash
# 使用新的 consumer group 从头消费
kafka-console-consumer \
  --bootstrap-server kafka:9092 \
  --topic seckill_order \
  --from-beginning \
  --group test-group-$(date +%s)

# 验证 Kafka 可正常收发
# 生产者
kafka-console-producer \
  --bootstrap-server kafka:9092 \
  --topic seckill_order
# 输入测试消息
```

### 问题 3：docker exec 报错 "container is not running"

**现象**

```
Error response from daemon: container xxx is not running
```

**根因**

- Kafka 容器已停止
- 历史日志仍存在，造成 "Kafka 还在运行" 的错觉

**解决方案**

```bash
# 1. 检查容器状态
docker ps -a | grep kafka

# 2. 查看崩溃日志
docker logs seckill_kafka --tail 100

# 3. 重启服务
docker compose down
docker compose up -d
```

---

### 问题 4：Docker 容器名称冲突

**现象**

```
Error: Conflict. The container name "seckill_kafka" is already in use
```

**解决方案**

```bash
# 方式一：停止并删除旧容器
docker compose down

# 方式二：删除指定容器
docker rm -f seckill_kafka
```

## 6. 服务部署与更新

### 6.1 查看各服务状态

```bash
# 所有容器
docker ps -a

# 单独查看
docker ps | grep kafka
docker ps | grep redis
docker ps | grep mysql
docker ps | grep app
```

### 6.2 完全重建（开发环境）

```bash
cd /home/way/seckill
sudo docker compose down -v       # 删除容器 + 数据卷
sudo docker compose up -d --build # 重新构建启动
```

> ⚠️ **警告**：`-v` 会清除 MySQL、Redis、Kafka 的所有数据。

### 6.3 保留数据重建（推荐）

```bash
sudo docker compose down          # 仅删除容器
sudo docker compose up -d --build
```

- ✅ MySQL 数据保留
- ✅ Redis 数据保留
- ✅ Kafka offset 保留

### 6.4 单独更新后端服务（高频操作）

```bash
# 仅重新构建 app 服务
sudo docker compose build app

# 仅重启 app 服务（不重启 MySQL / Redis / Kafka）
sudo docker compose up -d --no-deps app
```

- ✅ 零影响更新后端代码
- ✅ 其他服务保持运行

### 6.5 滚动更新（零停机）

```bash
# 启动新版本（不停止旧版本）
docker compose up -d --scale app=2 --no-recreate

# 确认新版本正常后，停止旧版本
docker stop seckill_app_old
docker rm seckill_app_old

# 恢复原配置
docker compose up -d --scale app=1
```

### 6.6 迁移到新机器

```bash
# 原机器：打包项目
cd /home/way
tar czf seckill.tar.gz seckill/

# 传输到新机器
scp seckill.tar.gz user@new-host:/home/user/

# 新机器：解压并启动
cd /home/user
tar xzf seckill.tar.gz
cd seckill
sudo docker compose up -d
```
